Order Lifecycle — QR dine-in and take-out

Customer scans QR → builds cart with size + sugar options → checks out → kitchen processes → payment collected. Customer phone polls every 5 seconds for status updates.

Figure 2 — Order lifecycle flowCustomer journey (left) · Kitchen/Staff workflow (right)
StatusTriggerWhoCustomer sees
NEWCustomer places orderAutoOrder confirmed, YANI-XXXX
PREPARINGStaff clicks STARTKitchen / Cashier“Your order is being prepared...”
READYStaff clicks READYKitchen / Cashier“Your order is ready!”
COMPLETEDStaff clicks COMPLETEAny role“Order completed. Thank you!”
CANCELLEDStaff clicks CANCEL (any time)Any roleOrder cancelled
Size + Sugar logic: Cold beverages show a size selector (Short ₱150 / Medium ₱170 / Tall ₱179) then a sugar selector (Grounded 25% / YANI 50% / Comfort 75% / Full Sweet 100%). Hot drinks skip size. Pastries and meals skip both. Choices stored per line item in size_choice and sugar_choice.
Service charge: 10% added automatically for DINE-IN only. Take-Out and platform orders are not charged. total = subtotal + (subtotal × 0.10).

Online Order Flow — Delivery and platform orders

Separate from the QR flow. Customers order via a web form without a table QR code. Orders go to the online_orders table and appear in admin under Platform Orders.

1
Customer visits /online-order
No QR scan required. Browses same menu items. No table selection.
2
Fills out delivery form
Name, contact number, delivery address or pick-up preference, and items. No size/sugar selectors on platform orders.
3
Submits via public API
Hits POST /api/online-order (no JWT required). Creates a row in online_orders with a unique order_ref.
4
Staff sees in Platform Orders tab
Admin panel → Platform Orders. Staff processes and updates status the same way as QR orders.
5
Payment on delivery or pickup
No GCash QR flow. Payment arranged separately (cash on delivery or counter pickup).
Key difference: Online orders use online_orders and online_order_items tables, NOT dine_in_orders. Excluded from dine-in revenue reports. No service charge applied. API endpoint: /api/online-order (separate from /api/pos).

Costing Module — Food cost for all 64 items

All 64 active items costed with real supplier pricing. Calculates gross margin and food cost % per item. Access: admin → Costing tab (OWNER login).

Figure 4 — Costing module flowIngredients → Recipes → Recipe Ingredients → Food Cost → Margin
1
Define ingredients
58 ingredients with cost per unit and unit type. e.g. “Espresso Powder, per gram, ₱0.45”
2
Create recipes
One recipe per menu item, linked by item_code. 64 recipes covering all active items.
3
Add ingredients per recipe
Specify quantity per serving — e.g. “Espresso Powder × 8g”. 301 ingredient-recipe links total.
4
System calculates food cost
cost_per_serving = ∑ (ingredient.cost_per_unit × qty)
5
Margin analysis
gross_margin = selling_price − cost_per_serving · food_cost_% = (cost / price) × 100
Flagged items: Katapang — current price is below cost, needs repricing to ₱220+. Affogato — high food cost due to Milk Ice Cream powder.

Auth & Roles — PIN → bcrypt → JWT → DB permission check

Staff enters PIN
  → bcrypt.compare(pin, pin_hash)  // rounds=12
  → signToken({userId, role})       // HS256, 4h expiry
  → Client stores in sessionStorage

Each write request:
  → verifyToken(bearer)            // decode JWT
  → getRolePermissions(action)     // DB lookup, 5-min cache
  → roles.includes(jwt.role)        // 403 if not allowed

Failed login:
  → failed_attempts++
  → At 5: locked_until = NOW() + 30min

Business Day Definition — 6:00 AM PHT to 5:59 AM PHT

Business day = 6:00 AM PHT → 5:59 AM PHT (next calendar day)
Orders placed between midnight and 5:59 AM are counted in the previous business day’s totals.
ScenarioResult
Order placed Thursday 11:00 PM PHTCounted in Thursday’s sales ✓
Order placed Friday 2:00 AM PHTCounted in Thursday’s sales (before 6AM cutoff)
Order placed Friday 6:01 AM PHTCounted in Friday’s sales ✓

Why 6AM? The cafe closes well before midnight. Orders after midnight are almost always test orders. The 6AM cutoff ensures daily totals match what staff think of as “yesterday’s business.” The shift summary counts COMPLETED + READY + PREPARING to avoid undercounting when staff skip the COMPLETE step.

Discount formula (printed receipts)

(total ÷ number of pax) × 20% × qualifying persons

Example: ₱526.90 total, 3 people, 1 PWD
  ₱526.90 ÷ 3 = ₱175.63 per person
  PWD discount = ₱175.63 × 20% = ₱35.13
  New total = ₱526.90 − ₱35.13 = ₱491.77

Tech Stack — Current production system

Frontend
HTML + Vanilla JS. No framework. 8 admin modules. Static files on Vercel CDN.
API
Vercel Serverless (Node.js). api/pos.js — 3,231 lines, 65 actions. pos.yanigardencafe.com
Database
Supabase PostgreSQL. 24 tables, RLS on all. ap-southeast-1 (Singapore).
Auth
PIN → bcrypt (12 rounds) → HS256 JWT (4h). 27 DB permission rules. 5-attempt lockout.
Email
Resend. Daily + weekly reports, low stock alerts.
Storage
Supabase Storage. payment-proofs bucket.
CI/CD
GitHub Actions. 15-check audit on every push.
Edge Functions
Supabase Deno. discount-integrity + low-stock-alert.
Known gaps to reach 90+: (1) Only 7/61 items have inventory tracking. (2) api/pos.js needs esbuild pipeline to split. (3) No uptime monitoring. (4) 13 GCash payments pending staff verification. (5) No receipt printer integration.
