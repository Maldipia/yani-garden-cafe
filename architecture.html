Database Sheets — Google Sheets as DB

The system uses a single Google Spreadsheet as its database. Each tab (sheet) acts as a database table with header-based column mapping. The backend reads and writes using the appendByHeaders() and updateByHeaders() utility functions.

🍵
YGC_MENU
Master menu: item codes, names, prices (Short/Medium/Tall), categories, active status
📋
ORDERS
Every order: ORDER_ID, table, status, subtotal, service charge, total, items JSON
📝
ORDER_ITEMS
Line-item detail for every order: item code, name snapshot, qty, unit price, line total
💳
PAYMENTS
Payment submissions: proof file ID, status (PENDING/VERIFIED/REJECTED), timestamps
👤
USERS
Admin accounts: user ID, name, role (OWNER/ADMIN/STAFF), hashed PIN, last login
📜
LOGS
Audit trail: every significant action with actor, target, details, timestamp, status
⚙️
SETTINGS
Global config: order prefix, service charge rate, business name, receipt settings
🪑
TABLES
Table registry: table numbers and their unique QR tokens for order validation

Data Flows — Key Journeys

Customer Order Flow

1

QR Code Scan

Customer scans table QR code → opens index.html?table=T1&token=abc123. Token is unique per table, stored in the TABLES sheet.

2

Menu Fetch

Frontend calls POST /api/pos with action: "getMenu". Vercel proxy forwards to GAS → reads YGC_MENU sheet → returns 38 active items as JSON.

3

Cart & Order Placement

Customer adds items to cart. On checkout, frontend calls placeOrder with cart items + table token. GAS validates token, calculates subtotal + 10% service charge, writes to ORDERS and ORDER_ITEMS sheets.

4

Live Dashboard Update

Admin dashboard polls getOrders every few seconds. New order appears instantly with status NEW. Staff updates to PREPARING → READY → COMPLETED.

Photo Upload Flow

1

Admin Selects Photo

In Menu Manager → Edit Item modal, admin clicks "Choose Photo" and selects a PNG/JPG file (max 5MB).

2

Upload to Vercel Function

Frontend encodes image as base64 and sends to POST /api/upload-image with item code and file extension.

3

GitHub Commit via API

Vercel function uses GITHUB_TOKEN to call the GitHub Contents API, committing the image to images/{CODE}.png in the main branch.

4

Auto-Deploy & CDN Live

GitHub commit triggers a new Vercel deployment (~1-2 min). Image is then live at https://yani-garden-cafe-d3l6.vercel.app/images/{CODE}.png on the global CDN.

🚀

CI/CD Pipeline

Vercel + GitHub Integration

Automatic Deployment on Every Push

The GitHub repository is connected to Vercel. Any commit pushed to the main branch — whether from a developer or from the photo upload endpoint — automatically triggers a new Vercel deployment. The production URL is updated within 1-2 minutes.

git push → auto-deploy Image upload → GitHub commit → auto-deploy GAS changes → manual redeploy required
⚠️

Important: Changes to Code.gs or any other Apps Script file require a manual redeploy in the Google Apps Script editor (Deploy → Manage Deployments → New Version). The Vercel CI/CD pipeline does not cover GAS deployments.

Security Model

🔑

PIN Authentication

Admin access is protected by a numeric PIN. The frontend hashes the PIN using SHA-256 before sending it to the backend. The backend compares against the stored hash in the USERS sheet — the raw PIN is never transmitted or stored.

🪙

Table Token Validation

Every customer order must include a valid table token. Tokens are unique per table, stored in the TABLES sheet, and validated server-side on every placeOrder call. Invalid tokens are rejected immediately.

🛡️

API Proxy Shield

The direct Google Apps Script URL is never exposed to the browser. All requests go through the Vercel /api/pos proxy, which handles authentication and shields the backend endpoint from direct access.

🚦

Rate Limiting

The backend implements an order throttle per table using the isThrottled() function. This prevents customers from accidentally or maliciously flooding the system with duplicate orders in a short time window.

🔒

Environment Secrets

Sensitive credentials (GITHUB_TOKEN, VERCEL_TOKEN) are stored as Vercel environment variables — never hardcoded in source files or committed to the repository.

📋

Audit Logging

Every significant action (order placed, payment verified, menu item edited, login) is written to the LOGS sheet via the logAction() function, creating a full audit trail with actor, timestamp, and outcome.
